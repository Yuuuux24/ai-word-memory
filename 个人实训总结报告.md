## 一、项目概述

本次实训项目"AI 单词记忆"是一个基于 **Next.js + Flask + Supabase** 的全栈 Web 应用。用户在应用中可浏览考研核心高频词汇（约 230 个）、查看 AI 生成的词根解析与趣味记忆口诀、记录学习历史、参与四选一闯关答题，支持用户注册登录和个人学习数据持久化。

整个开发周期为 5 天，共完成 43 轮 Git 提交，覆盖前端 4 个独立页面、后端 5 个蓝图模块共 15 个 REST API 接口，以及数据库 5 张表的完整设计。

## 二、遇到的最大困难及解决过程

开发过程中遇到的最大困难是 **Supabase 数据库连接兼容性问题**。

在项目 Day2 阶段，配置好 Supabase 密钥后启动后端始终无法获取单词数据。经过排查发现 Supabase 官方更新了密钥格式（从旧版 `eyJhbGci...` 长 JWT 改为 `sb_publishable_` / `sb_secret_` 前缀的新格式），而我使用的 `supabase-py` SDK 版本在 2.15 以下不支持新格式密钥。

**解决过程**：
1. 先尝试直接换回旧格式密钥，发现 Supabase 项目已不支持旧格式
2. 查阅 Supabase Python SDK 的 GitHub Release Notes，确认 >= 2.15 版本支持新密钥
3. 升级 `requirements.txt` 中的依赖版本约束为 `supabase>=2.15`
4. 重新 pip install 后成功联通数据库，获取到 4 个测试单词

另外，**JWT 鉴权改造**也是一大挑战。原项目所有接口零鉴权，引入 JWT 后需要对前后端进行系统性改造：后端新增 `utils/auth.py` 实现 token 生成、`@jwt_required` / `@optional_auth` 装饰器；前端创建 `utils/auth.js` 管理 token 存取和自动附带 Authorization 请求头；所有涉及用户数据的接口逐一添加鉴权校验。改造涉及 8 个后端文件 + 4 个前端文件，过程中还发现了分页计数 bug 和 AI 路由代码重复等问题一并修复。

## 三、对 AI 辅助编程的心得体会

### 3.1 AI 显著提升了开发效率

在 5 天时间内完成从零到一的全栈项目，AI 工具（CodeBuddy）发挥了关键作用：

- **生成代码骨架**：一次性生成 Flask 蓝图结构、Next.js 页面布局、Ant Design 组件模板，节省大量手工搭建时间
- **快速定位问题**：Supabase 密钥兼容性问题，AI 直接给出了升级依赖版本的解决方案
- **Code Review 质量保障**：AI 自动扫描 23 个源文件，发现 13 个问题（P0/P1/P2），包括代码重复、零鉴权、分页 bug、XSS 风险等，相当于有了一位免费的 Code Reviewer

### 3.2 AI 不能替代人的判断

在使用 AI 的过程中，我也深刻体会到：

- **需要拆分清晰的 Prompt**：一次性丢大段需求给 AI 往往输出质量差。将需求拆解为"搭建前端骨架 → 实现接口 → 联调 → UI 美化"等小步骤，每次只让 AI 做一件事，效果最好
- **AI 生成的代码需要审查**：AI 会生成重复函数定义（5 个文件各有自己的 `json_response`）、产生逻辑 bug（分页 total 计数错误）。如果没有 Code Review 环节直接交付，项目质量堪忧
- **架构决策需要人来拍板**：技术选型（Next.js vs Vue、Supabase vs MySQL）、JWT 实现方式、数据表设计等架构层面的决定需要开发者自己的理解和判断，AI 只能提供建议

### 3.3 迭代式开发 + AI 是最佳实践

每天一个明确的开发目标，通过多轮 Prompt 迭代推进：
1. 写 Prompt 描述需求 → 2. AI 生成代码 → 3. 测试运行 → 4. 反馈 bug 给 AI 修复 → 5. Git 提交
这个循环在整个开发周期中重复了 16 次，每次都能产出高质量的 commit。
